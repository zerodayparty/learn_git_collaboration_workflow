# 🟩 Pull Request와 GitHub Flow  

## 🟢 개념 개요  

### 🟡 Pull Request (PR) 및 GitHub Flow란 무엇인가  
팀 협업 시 프로젝트의 메인 코드(main 브랜치)에 안전하게 소스코드를 반영하기 위해 GitHub 등의 웹 서비스에서 제공하는 제안 및 검토 시스템과 그 협업 절차를 의미한다.  
- `PR` = Pull Request (내 변경 사항을 가져가서 병합해 달라고 요청함 / 풀 리퀘스트)  
- `Flow` = Flow (흐름, 절차)  
- 해설: 여러 개발자가 한 저장소에서 작업할 때, 승인 없이 아무나 메인 코드를 수정하여 시스템을 망가뜨리는 일을 방지하기 위해 사용된다.  



<br><br>

## 🟢 GitHub Flow의 표준 5단계 협업 절차  
가장 직관적이고 널리 쓰이는 브랜치 기반 협업 방식이다.  

### 🟡 1단계: 신규 기능용 브랜치 만들기  
- 메인 코드(main)는 항상 정상 작동하는 상태여야 하므로, 새로운 기능을 만들 때는 독립된 브랜치를 생성하여 작업한다.  
- 터미널 명령어: `git switch -c feature/login`  

### 🟡 2단계: 코드 수정 및 커밋 올리기  
- 브랜치 내에서 열심히 작업한 후 커밋을 생성하고 원격 저장소로 업로드한다.  
- 터미널 명령어: `git push origin feature/login`  

### 🟡 3단계: Pull Request (PR) 생성하기  
- GitHub 웹사이트에 접속하여 "내가 feature/login 브랜치에서 로그인을 다 구현했으니, 이 내용을 main 브랜치에 합쳐(Merge) 주세요"라고 의견(PR)을 올린다.  
- `Pull Request` = Pull (내 코드를 당겨서) + Request (병합을 요청하다)  

### 🟡 4단계: 코드 리뷰 및 의견 교환  
- 팀원들이 PR에 올려진 소스코드 변경 내역을 돋보기로 보듯이 검토하며 줄 단위로 피드백을 적는다.  
- 수정 보완 사항이 있으면 추가 커밋을 올려 PR 내용을 계속 업데이트한다.  

### 🟡 5단계: 병합(Merge) 및 브랜치 삭제  
- 모든 팀원의 검토와 동의를 받으면 메인 관리자가 GitHub 웹에서 'Merge Pull Request' 버튼을 눌러 소스코드를 main 브랜치에 최종 반영한다.  
- 합치기가 완료되면 쓸모 없어진 `feature/login` 원격 브랜치를 삭제하여 정리한다.

## 🟢 Pull Request 단어의 재미있는 비밀  
- 내 변경 사항을 '밀어 넣는' 것인데 왜 Push Request가 아니라 **Pull Request (가져가 달라는 요청)**일까?  
- 이는 오픈소스 협업 방식에서 유래했다. 메인 프로젝트의 관리 권한이 없는 외부 개발자가 "내가 코드를 유용하게 고쳤으니, 당신의 프로젝트로 내 코드를 **당겨가서(Pull)** 합쳐 달라"라고 관리자에게 부탁(Request)하는 격식 있는 요청이기 때문이다.  

<br><br>

## 🟢 직관적인 작동 비유  
Pull Request와 코드 리뷰는 대학 조별 과제 시 **최종 보고서를 제출하는 결재 과정**과 같다.  
- **내 방 노트북 (Local Branch)**: 보고서 초안을 열심히 작성하는 공간이다.  
- **조장 메일함에 초안 발송 (Push)**: 조장에게 검토용 파일을 보낸 상태이다.  
- **단톡방 검토 요청 (Pull Request)**: "팀원들아, 내가 1장 보고서 썼는데 확인하고 이상 없으면 최종 합친 보고서 파일(main)에 합쳐 줘"라고 단톡방에 보고서 링크를 올리는 행위이다.  
- **팀원 피드백 (Code Review)**: "여기에 오타가 있어", "표 형식을 맞춰 줘" 하고 검토해 주는 과정이다.  
- **최종 보고서 인쇄 (Merge)**: 수정 검토가 완료되어 정식 조별 과제물로 최종 확정하여 묶어두는 단계이다.
