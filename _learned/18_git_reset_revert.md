# 🟩 git reset, revert 명령어  

## 🟢 명령어 개요  

### 🟡 git reset, revert란 무엇인가  
과거에 잘못 만든 저장(Commit) 버전이나 실수한 코드들을 작업 이전 상태로 취소하고 되돌리기 위해 사용하는 버전 복구 명령어들이다.  
- `reset` = reset (재설정하다, 초기화하다)  
- `revert` = revert (되돌리다, 복원하다)  
- 해설: 이미 발생한 실수 코드를 안전하게 지우거나, 기록은 남겨둔 채 특정 시점의 코드로 회귀시키는 중요한 역할을 수행한다.  

<br><br>

## 🟢 주요 사용법과 단어 풀네임  

| 명령어 | 옵션 및 단어 풀네임 | 해설 |  
| :--- | :--- | :--- |
| `git reset --soft [커밋ID]` | `--soft` = soft (부드러운, 온건한) | 지정한 커밋 시점으로 이력을 되돌리지만, 취소된 파일 변경 내용은 버리지 않고 스테이징 영역(Stage)에 그대로 남겨두어 바로 다시 커밋할 수 있게 한다. |  
| `git reset --mixed [커밋ID]` | `--mixed` = mixed (혼합된, 기본 옵션) | 이력을 되돌리고 스테이징 영역도 비우지만, 실물 파일(작업 디렉터리)은 그대로 살려두어 코드 수정 상태로 놔둔다. |  
| `git reset --hard [커밋ID]` | `--hard` = hard (단단한, 가혹한) | 지정한 커밋 이후의 모든 기록과 실물 파일 변경 내역을 **완전히 삭제하고 깨끗이 날려버린다.** (가장 위험한 옵션) |  
| `git revert [취소할 커밋ID]` | `revert` = revert (되돌리다) | 대상 커밋의 수정 내용을 정반대로 뒤집는 **'새로운 취소 커밋'**을 한 단계 추가한다. 이전의 역사 기록들은 그대로 유지된다. |  

<br><br>

## 🟢 reset과 revert의 결정적인 차이  
두 명령어의 가장 핵심적인 차이는 **역사(History)를 지우는가, 유지하는가**에 있다.  

### 🟡 git reset (과거를 지우고 타임머신 타기)  
- **원리**: 시간을 과거로 통째로 돌려, 지정한 커밋 이후의 기록들을 마치 세상에 없었던 일처럼 완전히 삭제해 버린다.  
- **주의점**: 이미 GitHub(원격 저장소)에 올려 공유한 커밋을 `reset`으로 지우고 강제로 올리면 다른 팀원들의 이력과 충돌이 나서 협업 저장소가 망가진다.  
- **적용**: 아직 원격 저장소에 밀어넣지 않은 나만의 **로컬 컴퓨터 작업 실수를 조용히 감출 때** 사용한다.  

### 🟡 git revert (실수를 인정하고 취소선 긋기)  
- **원리**: 과거 기록을 단 하나도 지우지 않는다. 대신 "이전의 A 커밋 기능을 취소한다"는 의미의 새로운 버전(Commit)을 맨 위에 한 칸 쌓는다.  
- **안전성**: 역사를 파괴하지 않으므로 원격 저장소에 이미 공유된 버전도 안전하게 취소할 수 있으며, 팀원들과의 충돌을 방지한다.  
- **적용**: **이미 GitHub에 올려서 팀원들과 공유한 버전을 취소해야 할 때** 필수적으로 사용한다.  

<br><br>

## 🟢 직관적인 작동 비유  
문서 작성 중 실수를 바로잡는 상황에 비유할 수 있다.  
- **git reset**: 문서 작성 중 실수한 단락을 **지우개로 깨끗이 지우거나 실행 취소(Ctrl+Z)를 눌러** 흔적을 없애는 행위이다.  
- **git revert**: 실수로 계약서를 잘못 작성했다. 이미 상대방에게 전송했으므로 지우개로 지울 수 없다. 대신 **"기존 계약서 제3조는 무효로 한다"라는 공문(새 취소 커밋)을 정식으로 한 장 더 작성하여 덧붙이는 행위**와 같다.
